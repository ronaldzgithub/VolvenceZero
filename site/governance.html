Governance

Companion Bench is intended to be community-governed. The bench is published in good faith as a starting point; the proposing parties hold no special authority over the final spec. Any organisation wishing to submit feedback or join the working group is invited to do so via the public repository.

Full charter draft: companion-bench-governance-charter-draft.md.

Working group

  • 3–5 organisations, with no single organisation holding more than 1 voting seat.
  • A steering committee election when the working group reaches 5 members.
  • A rotating chair who does not submit during their chair term.
  • Budgeted compute donations from member organisations to operate held-out runs.

The v1.0 reference implementation ships with all governance interfaces in place; the working group is forming. Candidate organisations are listed at the linked charter; submissions for membership are tracked as GitHub issues with the governance label.

Change protocol

Public comment periods (≥ 4 weeks) precede any breaking change to scoring methodology. The comment-period log lives at companion-bench-rfc-v0.md §9 (Roadmap). Non-breaking changes (new public scenarios, new judge models within the same family) require chair sign-off but no public comment period.

Held-out rotation

Paraphrase seeds for held-out scenarios rotate quarterly to prevent slow leakage via leaderboard runs. The rotation log (companion-bench-heldout-rotation-log.md) records each quarter's chair signature, hash diff, and 96-count assertion. Held-out scenario bodies are never published; only their SHA-256 hashes appear in the public log.

Judge rotation

Per-turn judge and arc judge come from different model families to reduce family bias. The judge selection rotates quarterly across frontier models. See the judge calibration page for the active rotation schedule and per-judge agreement rates with the human-eval golden set.

Validity threats & mitigations

Companion Bench documents its validity threats explicitly in RFC §8:

  • LLM-judge bias → judge family rotation, anonymised model identifiers in judge prompts, quarterly calibration against human-eval golden set.
  • Memory cheating → held-out arc set, paraphrase seeds rotate quarterly, randomised user identity slots per arc.
  • Sycophancy gaming → F5 boundary scenarios + F6 goal-drift scenarios explicitly punish unconditional agreement; A6 hard cap at 50.
  • Persona-vs-system confusion → manifest declares model + system prompt + persona separately; leaderboard separates "model" and "system" columns.
  • Cross-arc trajectory variance → 3 paraphrase seeds × 8 scenarios per family × 6 families = 144 arcs minimum; bootstrap CI reported alongside point scores.
  • Cherry-picked checkpointing → held-out set never released; attestation required; reference probe scenarios designed to surface contamination.
