## A. 架构与概念澄清

### Q1 · "工业级认知运行时底座"到底是什么东西——是中间件、框架、库还是 SaaS？能不能用一张部署图说清楚？

#### 对方为什么会问

这是技术带头人的**第一反应式提问**。他们每天都在被各种 AI 新名词轰炸，听到"认知运行时底座"这种词会本能地警觉——是不是又一个套壳？是不是 LangChain 的换皮？所以**必须在 30 秒内用他们听得懂的工程语言定位清楚**，否则后面所有讲的东西都会被打折。

#### 应答策略

- **类比先行**：用"数据库之于业务系统""JVM 之于 Java 应用"这种他们秒懂的类比定位
- **讲形态、讲边界、讲对接面**——不讲内核算法
- **现场可以在白板上画一张最简部署图**

#### 应答稿（现场复述版）

"这是个非常好的问题，我用工程语言来回答。

**一句话定位**：它是一套**私有化部署的中间层服务**，形态上类似"**AI 助手的操作系统**"——部署时是**一组独立进程 + 一组 API + 一份状态存储**，中心现有的"智能预报操作系统"通过**标准 API 调用它**，它再去编排底层的瀚海智语 / 通义 / 深度求索等基座。

**一张最简部署图**：

```
[中心现有业务层：预报产品 / 会商 / 应急 / 公众服务]
                    ↓  标准 HTTP/gRPC 调用
[瀚海智语智能预报操作系统 + 智能体编排层]  ← 中心已建
                    ↓  标准 HTTP/gRPC 调用
┌──────────────────────────────────────────────┐
│   工业级认知运行时（我们这一层，私有化部署）      │
│   ├─ API 网关（对上：统一的会话 / 语境 / 追溯 API）│
│   ├─ 控制器服务（管"怎么学、怎么切换")            │
│   ├─ 记忆子系统（多时间尺度 / 可追溯 / 可回退)    │
│   ├─ 评估与证据子系统（出证据包、出审计日志)       │
│   └─ 契约总线（多 AI 之间的共同语境协议栈)        │
└──────────────────────────────────────────────┘
                    ↓  适配层
[基座大模型：瀚海智语 / 通义 / DeepSeek …]  ← 中心核心资产
[妈祖数值模式 / 算力]  ← 中心核心资产
```

**关键界定，打消三个可能的顾虑**：

1. **它不是新大模型**——不自带参数、不做训练、不抢占算力主导权
2. **它不是 Agent 框架替代品**——瀚海智语的编排层仍然是调度者，我们只是被它调用的"能力层"
3. **它不是一个黑盒 SaaS**——全部私有化部署，代码以二进制交付、关键组件可托管第三方触发释放源码

**技术形态上**的进一步说明：

- **语言栈**：核心服务 Python + 部分性能敏感模块 Rust/Go，与主流国产 AI 基础设施天然兼容
- **部署形态**：容器化（兼容 OpenEuler / Kylin 内网镜像仓库），**不依赖任何外部网络**
- **存储依赖**：标准 PostgreSQL + 对象存储（MinIO / 中心自有 S3），没有异构依赖
- **资源量级**（单实例初始规模）：约 **2–4 张国产加速卡 + 64–128 GB 内存 + 数 TB 对象存储**——远小于基座大模型的量级，是真正的"中间层"
- **和瀚海智语的对接面**：一组**幂等的 REST/gRPC API**，就这么多

所以最准确的比喻其实是：

- **它对中心的业务就像"数据库"**——你不会去和数据库抢应用层的事情，但所有需要"持久、一致、可追溯"的状态都最终归它管
- **它对大模型就像"JVM"**——JVM 不是 Java，但 Java 程序要跑得稳、跑得久、跑得可监控，就得靠 JVM

NDA 之后，我们可以把完整的模块清单、API 列表、部署拓扑逐项对齐到中心的运维规范。"

#### 核心佐证点（准备好可以展示的证据）

- 部署拓扑图（带真实组件名、端口、资源量级）
- 与 OpenEuler / Kylin 容器化镜像的兼容性声明
- 两个已落地领域中**同一个内核运行时的架构对比图**（证明"同一个内核"不是话术）

#### 可能的追问与应对

| 追问 | 应对要点 |
|---|---|
| "那它是不是 LangChain / LangGraph 的二次封装？" | 明确说**不是**。LangChain 是编排 DSL，在我们的**上一层**。我们这一层是它底下的"长期状态 / 内化 / 追溯 / 语境"。**它的优化目标是一次 prompt 链路跑通；我们的优化目标是 6 个月后还在跑稳**。 |
| "那它和 Databricks / 华为 ModelArts 这种平台是什么关系？" | 那些是**模型训练与部署平台**，关注"怎么把模型部署上去"。我们关注"**部署完之后在业务里怎么活下去**"——不是一层，不竞争。 |
| "是不是开源的？依赖什么开源组件？" | 核心内核**不开源**。依赖清单全部是主流开源件（PostgreSQL / Redis / MinIO / PyTorch 等），**不依赖任何境外云服务**，可以给中心提供 SBOM（软件物料清单）审计。 |
| "那你们这层挂了怎么办？会不会拖垮业务？" | 设计上是**旁路增强**而非强依赖——`bypass_mode` 打开后，流量直接透传到基座，业务不中断。这是我们在情感陪伴场景里运行 X 个月的硬约束。 |

### Q2 · 和 RAG / 长 context / LoRA / 全参微调 / Agent 框架的本质区别到底在哪？别用"第四条路"这种话，我要具体的技术差异

#### 对方为什么会问

这是算法负责人的**反话术提问**。deck 里的那张"四条路"对比表他们会当场挑刺——要么让你承认就是以上三条路的组合，要么让你说出一个他们之前没想过的维度。

**真正的区别维度是"变化发生在哪一层、持续多久、能不能回退"**——而不是"用的什么损失函数"。

#### 应答策略

- **不要绕**，直接上对比维度
- **承认每条路各自的价值和局限**（避免显得在贬低同行）
- **用一张"变化维度"的二维图**把区别立住——这一层 deck 里没有，属于现场加分项

#### 应答稿（现场复述版）

"这个问题问到点子上了。我不用'第四条路'这种抽象说法，我用一个具体的**维度切分**来回答。

一个 AI 系统在运行中，**可以在五个层面发生变化**：

| 层级 | 变化发生在 | 代价 | 可否回退 |
|---|---|---|---|
| L1 提示词层 | 每次请求的 prompt | 零 | 天然可回退 |
| L2 上下文层 | 上下文窗口 / RAG 召回 | 存储 + 延迟 | 天然可回退 |
| L3 流程层 | Agent 工作流 / 规则脚本 | 工程改动 | 版本管理即可 |
| **L4 控制器层** | **基座之上的薄适应层** | **中等（我们的位置）** | **可以设计成可回退** |
| L5 基座参数层 | 大模型权重本身 | 海量算力 + 稳定性 | 极难回退 |

**主流三条路分别在做什么**：

- **RAG / 长 context / 提示词工程**：只在 **L1–L2** 动，**永远不修改模型判断倾向**。知识被'读到'，下次对话还要重新喂。就像给考生每次都发一张小抄——考生本人**没有变化**。
- **LoRA / QLoRA / SFT / RLHF / 全参微调**：直接动 **L5**。每次学习都是在改大模型本身——好处是真的改了，坏处是：需要海量标注、稳定性难控、多任务互相打架、回退极其昂贵、一出问题整个模型都要回滚。
- **Agent 框架**（LangGraph / AutoGen / Dify / 华为 ModelEngine）：在 **L3** 做编排——工作流是**人工预设的**，AI 的长期'状态'仍然要么放 prompt（L1）要么放记忆数据库（L2）——没有一个真正长期持有、可演化的内部结构。

**我们在 L4 这一层**，这一层有两个业界没有充分利用的特性：

1. **它足够小所以可以真正在线学习**——不会破坏稳定基座
2. **它又足够深所以能承载判断倾向**——不是简单的'往 prompt 里塞几句话'

换句话说：

- **RAG 是'查资料'**——别人的知识，当下临时借用
- **微调是'做脑手术'**——风险大、不可逆、要海量样本
- **Agent 是'写 SOP'**——固定流程，无法应对没见过的情况
- **我们是'带学徒'**——跟着老师傅跑几个月，师傅的直觉就慢慢传到学徒的**判断模式**里；师傅不在场的时候，学徒也能做出'师傅会做的判断'

这就是为什么 deck 里列的三件事——经验内化 / 少量交互 / 多 AI 共同语境——**只能**在 L4 这一层做。

- 在 L1–L2 做：每次对话重喂，永远内化不了
- 在 L5 做：代价太大、稳定性不可控，而且**模型权重不可回退**直接违反等保要求
- 在 L3 做：工作流写死，应对不了气候分布漂移这种变化

**最后一句定位**：主流三条路和我们**在做的事不在同一个维度上**。我们甚至可以**叠在他们之上跑**——比如中心完全可以继续用瀚海智语做基座、用 LangGraph 做编排、用 RAG 做冷启动，我们只是给这个已有栈**补上'跑得久、跑得稳、越跑越好'那一层**。"

#### 核心佐证点

- **变化维度对比表**（上面那张五层表，可做成 A4 打印件递出去）
- 在两个已落地领域中，**L4 状态变化曲线图**（横轴时间，纵轴判断一致性）——证明 L4 是真在"变化"，不是没变
- `bypass_mode` 切换时的 A/B 对比数据——证明我们这一层是**可插拔的**，不是强依赖

#### 可能的追问与应对

| 追问 | 应对要点 |
|---|---|
| "在 L4 搞这么多东西，是不是就是复杂版的 LoRA？" | **不是**。LoRA 的目标函数是拟合训练集，是**单次训练一次性套上的适配层**；我们这一层是一个**在线演化的控制系统**，目标是让基座在长期业务里表现稳定——两者在**优化对象、时间尺度、失败模式**上都不一样。Q3 会详细讲。 |
| "那为什么业界不都这么干？" | 两个原因：**一、学术上是个较冷门方向**，不是 Transformer 这种热点；**二、工程上要做好这一层需要跨几门学科**——主动学习 + 非平稳学习 + 在线控制 + 分布式一致性——大厂的 AI 组通常只懂前两样。这也是为什么我们科学家团队 15 年都在深耕这个方向，而不是追热点。 |
| "OpenAI 的 memory、Anthropic 的 projects 不都是 L4 吗？" | 它们是 **L2 变种**——把 RAG 召回的内容格式化成'记忆'再塞回 prompt，**本质上还是长 context 工程**。不改变模型的内部判断倾向，模型随时可以'忘记'这些 memory（只要不召回）。**我们的 L4 是持久化的内部状态**，不靠召回。 |
| "你们是不是就是把 MoE 套壳卖？" | 不是。MoE 是 L5 的路由机制，训练好就固定了。我们在 L4，运行时**可学习、可回退、可追溯**，和 MoE 的路由是不同事情。 |

### Q3 · PPT 里提到"在通义千问开源权重上的残差干预机制"——这具体是什么技术？和 LoRA / Adapter / Prefix-tuning 的关系是什么？

#### 对方为什么会问

这是 deck 里**唯一一个具体技术名词**，懂 Transformer 的技术人员会立刻问。这个问题也是验证"是不是真有技术"的试金石——答不上来或者和 LoRA 混为一谈，前面讲的一切可信度直接打折；答得清楚，对方反而会更踏实。

**重要**：这个问题**可以讲到概念层**，但**不要讲到公式层 / 代码层**。概念层足以建立专业感，公式层留到 NDA 后。

#### 应答策略

- **先定位家族**：这属于"推理期控制"（inference-time control）而不是"离线微调"家族
- **把差异讲清**：和 LoRA/Adapter 的根本差异是**改变发生的时刻和对象**
- **落到关键优势**：基座权重字节级不变、每次干预可审计可回退

#### 应答稿（现场复述版）

"好，我讲到概念层，细节留到签了 NDA 之后。

**先把家族分清**：围绕大模型的适应技术，过去几年分成两大家族：

**家族一 · 离线适应家族**——在**训练阶段**改模型：
- **LoRA / QLoRA**：在注意力矩阵上加低秩分解 ΔW = BA，要训练、要梯度、要标注样本
- **Adapter**：在 Transformer 层之间插入小型瓶颈 MLP，同样要训练
- **Prefix-tuning / Prompt-tuning**：在输入 token 前面加一段可学习的 soft prompt
- **SFT / DPO / RLHF**：全参或大部分参数的训练

**共同特点**：训完就固定，部署后不再变，改需要重训。

**家族二 · 推理期控制家族**（activation steering / representation engineering 家族）——在**前向推理过程中**改激活值：
- **代表工作**：Anthropic 的 activation steering、MIT 的 ROME / MEMIT、NeurIPS 这两年关于 'steering vectors' 的一系列论文
- **共同思路**：在 Transformer 的**残差流（residual stream）**上，根据当前上下文注入一个**可学习、可条件化、可回退的小向量**
- **关键特征**：
  - **基座权重字节级不变**（md5 / sha256 完全一致）
  - 干预是**上下文相关的**——不同场景注入不同的干预
  - 每次干预**可计量、可记录、可回放**

**我们在家族二里**，再向前推了两步：

**第一步 · 从静态 steering 到在线自适应**：
业界现有的 steering vector 都是**离线训练出的静态向量**——场景 A 用向量 a、场景 B 用向量 b。我们把它做成了**一个在线的、基于控制理论的自适应系统**——**干预本身会根据预测误差和反馈信号持续调整**。这就是为什么它能'学习'——不是改基座，而是改'干预策略'。

**第二步 · 从单次干预到多时间尺度的嵌套干预**：
一次前向里有多个时间尺度的干预在叠加——**快的部分适应当前对话**（秒级）、**中间的部分适应当前会商周期**（天级）、**慢的部分沉淀为长期判断倾向**（月级）。这三层互相约束，慢层不会被快层轻易冲掉——这正是'经验内化'的工程基础。

**和 LoRA / Adapter 的本质区别总结成一句话**：

- LoRA / Adapter 改的是**模型权重的增量**，部署后是**静态适配器**
- 我们改的是**每次前向推理时残差流上的条件化干预**，部署后是**动态控制器**

**对中心的实际意义**：

1. **瀚海智语的权重字节级不变**——中心的基座模型文件、所有版本管理、所有合规审批都不受影响
2. **中心可以随时切断我们的干预** —— `bypass_mode` 一开，中心拿到的就是一个原版瀚海智语
3. **每一次干预都是一条可审计的记录**——写进证据链，满足等保可追溯要求
4. **切换基座只需要适配残差接口**——通义千问上验证过的机制，迁到瀚海智语主要是适配工作，不是重新发明

我们在**通义千问开源权重上的完整证据包**已经存档，包括：
- 基座权重前后 sha256 一致性校验
- 干预向量的完整日志
- 开关打开 / 关闭的 A/B 对比数据
- 跨随机种子的稳定性曲线

这份证据包，**签了 NDA 我们现场打开给你们看**。"

#### 核心佐证点

- **通义千问残差干预完整证据包**（内部已存档）
  - 基座权重文件 sha256 校验报告
  - 干预日志示例
  - `bypass_mode` A/B 对比曲线
- **两个领域同一干预机制的对比图**（证明机制领域无关）
- **公开论文列表**：Anthropic activation steering、ROME、MEMIT——作为"家族定位"的参考文献，显示我们站在严肃的学术传承上而不是拍脑袋发明

| 追问 | 应对要点 |
|---|---|
| "干预向量是怎么学出来的？用什么损失函数？" | "这涉及我们三项原创理论方向之一的具体实现，**签了 NDA 可以深度交流**。今天只讲：它**不是标准的交叉熵损失**，因为我们要同时优化'少样本拟合'和'分布鲁棒性'两个目标，这正是杨柳博士《分布漂移下的主动学习》《可靠的主动学徒学习》两篇工作的延伸。" |
| "干预幅度有没有上界？会不会放飞到把瀚海智语的回答搞乱？" | 有**硬上界**。干预向量有显式范数约束（L2 ball），并且有**契约守卫**（contract guard）——如果干预导致输出违反业务规则（例如预警级别与浪高对应关系），系统**自动回退到 bypass**。这是我们在情感陪伴场景做了几万次会话后沉淀的边界机制。 |
| "这机制在昇腾 / 寒武纪上能跑吗？CUDA 算子移植过吗？" | 残差干预是**张量加法 + 矩阵乘法**，是最基础的算子——**没有任何特殊算子依赖**。昇腾的 CANN、寒武纪的 MLU 都支持标准 PyTorch 前向 hook，适配工作量远低于移植 LoRA 训练。 |
| "通义千问是稠密模型，瀚海智语可能是 MoE，残差流结构不一样怎么办？" | **好问题**。MoE 的残差流在专家路由之前仍然存在，我们的干预点可以挂在**专家选择之前的共享残差流上**，架构上是兼容的。具体适配量视瀚海智语内部结构而定，**深度交流阶段可以一起看代码对齐接入点**。 |

## B. 专家经验内化

### Q4 · 你们又说不改基础模型权重、又说能让专家经验"真正内化"——逻辑上矛盾吧？到底怎么做到？"内化"和"放在 RAG 里被读到"的具体差别在哪？

#### 对方为什么会问

这是 deck 里**最容易被打的一个宣称**——"内化"这个词很漂亮，但技术上听起来像宣传。对方算法专家的直觉是：不改权重就是没动模型，没动模型怎么叫'内化'？

**必须在 3 分钟内用一个业界专家听完会点头的定义把这个宣称立住**，否则 deck 里的核心价值主张（少量交互沉淀几十年经验）全部空心化。

#### 应答策略

- **先把"内化"做成一个可测量的定义**，不是文学修辞
- **用三档对比**把"读到 / 背下 / 内化"拉开
- **拿两个已落地领域的具体现象做佐证**
- **明确告诉对方"内化"的物质基础是什么**——不是改基座，而是累积在 L4 控制器层

#### 应答稿（现场复述版）

"好，我先承认'内化'这个词听起来确实像营销话。我给出一个**可测量的严格定义**，然后你们判断我们做到没做到。

**我们定义的'内化'是一个可验证的行为标准**，不是比喻：

> 一条经验被'内化'，意味着——**在不给出该经验对应的提示词、不在 context 里重复该经验、RAG 也不召回该经验的情况下，AI 对相关输入仍然做出符合该经验的判断。**

这是一个**可以被客观验证**的标准——**把 context 清空、把 RAG 关掉、只给 AI 一个全新的输入**，看它还能不能按专家风格判断。能，就是内化了；不能，就不是。

**对照三档来看**：

| 层次 | 触发方式 | 断网测试结果 | 典型技术 |
|---|---|---|---|
| **读到** | 每次重新塞进 prompt | 清空 context 后**完全消失** | 提示词工程、长 context |
| **背下** | 向量召回后塞进 prompt | 清空 context 但保留向量库：**还能用**；两个都清：**消失** | RAG / 知识图谱 |
| **内化** | 不塞、不召回，AI 自己出 | 清空 context、清空向量库：**仍然表现出该经验** | 我们 |

**'内化'的物质基础在哪儿**？——在 L4 控制器层的**慢时间尺度参数**里。

前面 Q3 讲到我们在残差流上有多时间尺度的干预。其中**慢时间尺度的干预向量**不挂在任何一次会话、任何一条 RAG 条目上，而是作为控制器层的长期状态。它的行为就像一个**低通滤波后的专家风格提炼**——具体某次会商的细节会被快时间尺度吸收掉，但**反复出现的判断倾向**（例如'该海区 2 月气压切变偏保守预警'）会被慢时间尺度缓慢吸纳，一旦吸纳进来，**即使原始样本、原始对话全都删除了，判断倾向仍然保留**。

这个机制和基座权重是完全解耦的——基座是冻结的教科书，控制器层是学徒的'内功'。

**工程上我们怎么证明这个机制确实在工作**：

在情感陪伴场景里，我们做过一个标准测试——把一位用户 3 个月的**全部对话历史删除**、**把召回索引清空**、**让他用一个新会话登录**。如果系统只是'读到'或'背下'，应该完全失忆；但我们观察到的是——**系统仍然以对齐该用户偏好的方式响应**（比如该用户偏好的语气节奏、话题切入方式、特定话题的回避风格）。这组对照实验有完整的盲评报告，NDA 后可以展示。

在编程结对场景里也一样——删除历史，AI 仍然按该工程师的代码风格、命名偏好、异常处理习惯写代码。

**对海洋预报的具体含义**：

- 当老张跟 AI 工作 3 个月后，**即使把 3 个月对话记录全删**，AI 面对'渤海湾 2 月气压切变'这种场景，**仍然会给出带有老张判断倾向的答案**——而不是教科书式回答
- 这才是中心几代人积累的'模式之上的认知层'**真正被结构化到系统里**的意义
- 这也是和 RAG / 知识图谱的根本差别——RAG 是'老张的著作放在图书馆里'，内化是'老张的直觉长在学徒身上'

**对'为什么不矛盾'的最后一句**：

- 基座权重不变 = 瀚海智语本身不变
- 但 AI 系统 = 基座 + 控制器层
- **真正在业务里跑的是'AI 系统'，它的判断倾向在变**
- **瀚海智语的稳定性、合规性、版本管理都不受影响，中心的自主可控不受影响**——这是我们承诺的工程约束

#### 核心佐证点

- **"断网测试"规程文档**（内部已有，可在 NDA 后展示）
- **两个领域的"删除 + 新会话"盲评报告**——证明'内化'可以通过断网测试验证
- **L4 慢时间尺度参数随时间变化曲线**——证明参数在真正漂移，不是静态的
- **三档对比表**（读到 / 背下 / 内化）——现场白板可画

#### 可能的追问与应对

| 追问 | 应对要点 |
|---|---|
| "那 L4 的慢参数其实就是一个'不挂在 RAG 里的 embedding'，本质不还是 RAG 吗？" | 不是。RAG 的 embedding 是**输入空间的索引**（用 query 去找类似内容）；我们的慢参数是**残差流空间的控制信号**（直接改变 AI 做判断时的内部表征）。前者影响'AI 看到什么'，后者影响'AI 怎么看'。这是**感知 vs 判断的区别**——举一个例子：RAG 是给医生翻病历，我们是让医生经年累月慢慢形成的临床直觉。 |
| "可是慢参数也是一组数字，它改变了，从某种意义上'模型'就变了。这不是自相矛盾吗？" | **基座模型字节级不变**是一个**合规 / 自主可控**意义上的承诺；**AI 系统整体在变**是一个**业务能力**意义上的事实。这两个描述不矛盾——就像 JVM 的字节码不变，但应用状态在变，这是两个层面的事情。 |
| "你们这个'断网测试'是不是其实是 prompt 已经在系统 prompt 里隐藏注入了？" | 可以现场**完全透明地**做——把系统 prompt、工具返回、RAG 召回全部开放审查，只允许用户 query 作为输入。这是中心任何一位算法人员都可以现场设计的对照实验，我们欢迎。 |
| "情感陪伴那种场景，'内化'了也没人验证对错。海洋预报要求真对得起业务判断——你们的内化会不会'学错了'？" | 非常好的问题，这是 Q6（避免过拟合）和 Q14（对齐客观指标）要回答的。核心机制是：**慢参数的更新要通过一个'门控'**（modification gate），只有经过多轮反馈、跨专家、跨案例都稳定体现的判断倾向才能沉淀。单个专家临时的、情绪化的、低置信的判断不会进入慢参数。具体机制 Q6 详细讲。 |

### Q5 · "主动学徒式学习"到底怎么和专家交互？老预报员每周 2 小时具体在做什么动作？别让老专家填表

#### 对方为什么会问

这是**业务侧副总 / 首席预报员最实在的问题**。他们每天在跟老专家打交道，知道老专家的时间**多么稀缺、多么不愿意被数字化流程打断**。任何需要老专家"标注几千条样本""每天写日志""每次做判断都点赞 / 踩一下"的方案，**进不了中心的门**。

这个问题答不好，再漂亮的技术都落不了地。

#### 应答策略

- **先承认这个约束的不可让步性**——老专家不会、也不该去做标注工人
- **给出一个符合专家日常工作节奏的交互模式**——"不打扰工作、只在自然会商点嵌入"
- **量化到具体动作、具体耗时、具体负担**
- **点出这和 RLHF / SFT 式的标注完全不同**

#### 应答稿（现场复述版）

"这是个非常实在的问题。我直接给答案：**老预报员不需要标注、不需要填表、不需要写额外报告**——这是我们的硬约束，也是'主动学徒'这个词的核心含义。

我们和传统 AI 训练的交互方式有本质不同：

| 范式 | 谁挑样本 | 专家负担 | 每 10 小时专家时间能产生的价值 |
|---|---|---|---|
| 传统 SFT | 运营团队大量标注 | 几百上千小时标注 | 几百条样本 → 训练出一个可能过拟合的模型 |
| RLHF | 运营团队做偏好排序 | 海量对比打分 | 一轮迭代可能要数万对比 |
| **主动学徒学习（我们）** | **AI 自己挑最关键的 10 个案例** | **专家只需点评这 10 个** | **10 条高信息样本 → 一次有意义的判断倾向更新** |

**核心机制**：**AI 主动挑差异大、不确定性高、有学习价值的案例**，**主动问专家**'这种情况你会怎么判、和我的判断差在哪'。

**每周 2 小时具体做什么**——我们设想的节奏（会结合中心实际值班制度调整）：

**场景一 · 日常值班影子跑（0 额外时间）**
- 试点期间，AI 在老专家值班时**并行影子运行**——老专家正常做预报
- AI 自己记录'如果是我我会判 X'，并和老专家最终判断对齐
- **这个阶段专家完全不被打扰、不做任何额外动作**

**场景二 · 每周 20 分钟"差异点评"（每周 1 次）**
- AI 每周从影子运行中**挑 5–10 个"我和你判得不一样的案例"**（主动学习的选样）
- 以简短卡片形式发给专家，一张卡片不超过 30 秒读完：
  - 当时实况
  - AI 的判断 + AI 认为自己不确定的点
  - 老专家的判断
  - 两条询问：**"我这里错在哪？是数据没看到、还是判断方式不对？"**
- 专家点选 + 一句话点评即可，**不需要写长文**
- 5–10 张卡片，约 15–20 分钟

**场景三 · 每月 1 次 40 分钟"会商复盘"**
- 重要天气过程结束后，AI 整理一次'我这一过程中的关键判断节点 + 和你判断的差异'
- 专家做一次**会商式复盘**——专家本来也要做复盘的
- AI 参与这次复盘、提问，**复盘本身是专家的本职工作**

**场景四 · 偶发"不确定触发"（按需）**
- 遇到 AI 高不确定性的**新型天气过程**（比如 10 年一遇的极端情境），AI 会**主动请教**老专家一句
- 这种 interrupt 我们设了上限——**每周不超过 2 次**，避免打扰

把这四种场景加起来，**老预报员每周在 AI 系统上的'额外时间'不超过 2 小时**，而且这 2 小时里**有 40 分钟原本就是他本职的会商复盘**，纯粹因 AI 而增加的时间大约 **1 小时/周**。

**这种交互范式为什么能做到 '少样本高价值'**：

- 背后是杨柳博士 2011 NIPS 和 2025 ALT 两篇工作的延伸——**主动学习的信息论下界**：样本 N 对一个分布的刻画精度，不是由 N 决定的，而是由 N 的'信息含量'决定的。AI 挑高信息量的样本，小 N 也能拟合好
- **专家的每一句点评是一个极高信息密度的标签**——不是'是 / 否'、不是'1-5 分'，而是'差异归因'：'这里我看了渤海近 30 年的异常封冻案例，这种气压切变 3 天内有 6 成封冻率'
- 这种'归因式标签'一条顶 RLHF 里 100 条对比

**和 RLHF / SFT 的根本对比**：

- RLHF / SFT：**量大、标签浅**——几十万条'好 / 不好'
- 主动学徒：**量少、标签深**——每周几十条'为什么'

**对中心的具体承诺**：

- 我们在 3 个月试点方案里**明确写入**：每位参与试点的老预报员**每周 AI 系统额外时间不超过 2 小时**
- 如果超过，试点中止条件之一触发
- 交互形式由中心业务处和专家一起定，我们只提供工具

#### 核心佐证点

- **差异卡片样例**（情感陪伴 / 编程协作领域的实际截图）——证明卡片确实是 30 秒可点评级别
- **每周专家时间消耗表**（两个领域的实际数据）
- **主动学习选样策略论文清单**（杨柳博士的方向）
- **5 条交互约束**写入合作协议（NDA 后可草稿化）

#### 可能的追问与应对

| 追问 | 应对要点 |
|---|---|
| "老预报员值班任务已经很重，影子跑期间会不会占用值班席位的资源 / 电脑 / 带宽？" | 不会。影子运行跑在**独立硬件 / 独立网段**，通过只读方式订阅值班席位的输出日志，**对值班席位零 IO 侵扰**。专家看到的就是正常值班环境。 |
| "老专家不愿意点评 AI 怎么办？尤其是担心 AI 学了自己的东西以后替代自己" | 三条对冲：**（1）**对专家透明展示'AI 学了什么、学到什么程度、哪些没学到'，让专家感受到自己的不可替代性；**（2）**AI 的定位是'值班助理 + 会商辅助'——明确说，不是替代，是延展专家能力的工具；**（3）**参与试点的专家在合作协议中**有署名权**——这是'杨柳团队 × X 专家'的联合研究成果。这个机制在情感陪伴场景帮我们解决了类似的用户抵触。 |
| "10 张卡片一周，够吗？3 个月才 100 多张，能学出什么来？" | 够。**这正是主动学徒学习 vs RLHF 的数量级差异**——因为我们选的是**信息量上界**的样本（最能分辨判断倾向的 case），而不是随机抽样。杨柳博士 2025 ALT 论文里有理论保证：**在可学性假设下，主动学徒的样本复杂度下界是 RLHF 的 O(log n) 量级**。工程上在编程结对场景我们验证过——**100 条左右的差异点评足够让 AI 对齐一位工程师的判断风格到业务可接受水平**。 |
| "如果专家点评本身就不一致呢？（今天说这样判、明天说那样判）" | 系统**不会在单次点评后立即更新慢参数**——见 Q6。只有跨案例、跨时段都稳定体现的倾向才会进入长期参数。临时的、矛盾的点评会留在快时间尺度里，不会污染长期判断倾向。 |
| "这种学徒方式，老专家有没有可能 '教错'——比如他个人的偏见被学进去了？" | 这正是 Q7 要回答的——**多专家并存 + 权威度加权**，避免单一专家偏见。同时**契约层**会做一层硬约束——任何违反业务规则（例如 TS 评分会明显下降）的'学徒结果'会被自动拦截。 |

### Q6 · 少样本学习最老生常谈的两个问题——过拟合和灾难性遗忘——你们怎么解决？每周 10 条样本，一个月几十条就想让 AI 判风浪，这在机器学习上站不住脚

#### 对方为什么会问

懂机器学习的人的**本能怀疑**——少样本 + 在线学习 = 过拟合 + 遗忘，这在学术界争了 30 年。

deck 里用"少量交互就能向专家学习"这种话，**对方算法专家一定会在这个点上找漏洞**。如果答不上来，前面的"内化"、"主动学徒"都会被质疑是营销。

**必须从"架构上本身就不走传统那条路"来解这个题**，而不是"用了什么正则化技巧"。

#### 应答策略

- **承认传统路径下这个顾虑是对的**
- **点明我们不走传统路径**——不是在基座上做少样本微调，这是根本
- **给出具体四层防御机制**
- **用两个已落地领域的实际数据说话**——不是理论保证，是实测结果

#### 应答稿（现场复述版）

"这是教科书级的好问题。**如果我们是做少样本微调，这两个问题确实无解**——学界争了 30 年没真正解决。**我们之所以能回避这个困境，是因为架构上根本不走少样本微调这条路**。

让我把四层防御机制讲清楚：

**第一层防御 · 架构层：基座权重完全不动**

这是最重要的一点。**灾难性遗忘的前提是'权重被新样本更新了'**——我们从根上不更新基座权重。瀚海智语基座里已经承载的所有世界知识、物理规律、语言能力，**不会因为任何学徒交互而丢失一点**。

换句话说：
- **传统 SFT 路径**：10 个样本 + 反向传播 → 可能把几百亿参数里某些重要部分覆盖掉 → 灾难性遗忘
- **我们的路径**：10 个样本 → 只进入 L4 控制器层 → 基座参数零变化 → **遗忘问题从架构上不存在**

**第二层防御 · 修改门控（Modification Gate）**

即使是 L4 控制器层的慢参数，也**不是'收到点评立刻改'**。每次学徒交互后，系统会计算一个**候选更新**，这个候选更新必须通过 4 项门槛才被采纳到慢时间尺度：

| 门槛 | 含义 | 没过的处理 |
|---|---|---|
| **跨案例一致性** | 该倾向在至少 K 个独立案例里重复出现 | 留在快时间尺度，**不进慢参数** |
| **holdout 不劣化** | 在预留评估集（中心历史签发 + 实况对照）上**业务指标（TS、漏报率等）不变差** | 候选更新**丢弃** |
| **稳定性** | 跨随机种子、跨上下文扰动仍稳定 | 候选更新**丢弃** |
| **契约合规** | 不违反硬业务规则（预警级别 vs 浪高对应表等） | 候选更新**丢弃** |

**过不了门的候选更新，不会污染长期参数**。

**第三层防御 · 多时间尺度分层**

我们有四层时间尺度（在线快 / 会话中 / 后台慢 / 离线重度），**快层不会直接写慢层**——慢层由后台反思引擎基于**已经过了门控的候选**做积分式更新。这就杜绝了'当天点评当天就把长期行为改飞了'的问题。

举一个具体对照：
- **传统在线学习**：专家今天说'该海区 2 月应保守预警'→ 梯度立刻进参数 → 隔天如果专家改口'这个海区其实不怕'→ 参数又被冲掉 → **震荡**
- **我们的机制**：该倾向进快时间尺度 → 如果下周其他案例和其他专家不支持 → 快时间尺度自然衰减 → **慢参数完全没受影响**
- 只有**反复、跨时段、跨专家、跨评估指标稳定**的倾向，才最终沉淀进慢参数

**第四层防御 · 负迁移检测与自动回退**

每次慢参数更新后，系统自动在**双对照集**上评估：
- **A 集**：新倾向相关的案例（应该变好）
- **B 集**：与新倾向无关的历史案例（**不应该变差**）

如果 A 集没变好 or B 集变差超过阈值，**自动回退到更新前**。这个机制对应 Q9 的可回退粒度——每一次慢参数更新都是一个可回滚的版本。

**理论基础来自杨柳博士两篇工作**：

- 2011 NIPS《分布漂移下的主动学习》——证明了在分布漂移条件下'少样本 + 选择性更新'可以有 PAC 级样本复杂度保证
- 2025 ALT《可靠的主动学徒学习》——证明了在学徒框架里**可靠性（reliable）** 可以在少样本下得到形式化保证

**工程上我们有实测数据**（两个已落地领域的数据，NDA 后可展示）：

- 情感陪伴：单用户 6 个月在线学习，**holdout 集性能单调不降**（跨 X 次学徒交互，0 次触发回退）
- 编程结对：100 条左右差异点评 + 后续 3 个月使用，代码风格一致性稳定提升，**通用代码能力无退化**（用标准 benchmark 测量）

**对海洋预报场景的具体承诺**：
- 3 个月试点中，中心可指定一个**holdout 评估集**（历史案例 + 实况对照）
- 每次慢参数更新后，**holdout 集性能必须不劣化**才允许生效
- 如果任何一次更新后在 holdout 集上业务指标（TS、漏报率、GS 等）下降超过阈值，**系统自动回退**
- 这条**写入合作协议**

#### 核心佐证点

- **修改门控 4 项门槛规程**（NDA 后完整披露）
- **两个领域 holdout 稳定性曲线**
- **自动回退触发记录**（证明机制确实在运行，不是纸面设计）
- **杨柳博士 2011 / 2025 两篇论文**（公开可查，PAC 保证）

#### 可能的追问与应对

| 追问 | 应对要点 |
|---|---|
| "Holdout 集是你们自己造的吗？里面数据你们选的吗？会不会是'演员测试演员'？" | 方法论承诺：**Holdout 集由中心选**——中心的质量体系已经有历史签发 + 实况对照数据集，**我们只消费、不生产**这个数据集。甚至我们可以对 holdout 集只知道输入 / 不知道标签（类似高考盲考）——评估由中心运行。 |
| "门控阈值谁定？你们自己定还不是想放就放？" | 阈值在**试点启动前签字前敲定**（五条承诺之一），具体值跟中心评估专家一起讨论、写入试点工作说明书。过程中**不能单方面调整**——如要调整必须双方书面同意。 |
| "多时间尺度听起来就是把在线学习分成了 N 个学习率不同的层——这不就是层级 Adam 吗？" | **不是**。层级 Adam 是**同一个梯度分到不同层按不同步长更新**；我们的时间尺度分层是**不同层级看到的训练信号不一样**——快层看单次交互、慢层只看通过门控的候选更新。本质上是一个**嵌套学习架构**（Nested Learning），而不是简单的层级学习率。杨柳博士团队在这一方向有持续研究。 |
| "你们既然自动回退，那万一回退触发很频繁，试点不就看不到任何改进吗？" | 有可能。这反而是**健康的结果**——说明当前交互量 / 交互质量不足以形成稳定内化。报告里会诚实披露，并分析是**数据量不够**、**交互形式不对**、还是**该领域本身需要更多交互**。这比'强行出效果'更有价值。 |

### Q7 · 一个海区有好几位权威预报员，张三对涌浪保守、李四对海冰激进——AI 最后到底学谁的？系统支持多"分身"共存吗？

#### 对方为什么会问

这是**资深预报员和业务副总联合提问**——既是技术问题也是**业务政治问题**：

- 技术上：不同专家判断冲突，小样本学习里必然遇到
- 业务上：哪位专家的风格"成为 AI 的官方版"？这是一件**内部权力敏感**的事
- 风险上：如果 AI 只学一位专家的风格，**等于把所有鸡蛋放在一个篮子里**——该专家个人的系统性偏差会被系统固化

这个问题回答得不好，**专家群体对项目的配合度会大打折扣**。

#### 应答策略

- **正面回答"支持多分身"**——不是折中、不是投票、不是取平均
- **技术上讲清多分身的机制**——独立慢参数痕迹、条件化调用
- **业务上讲清价值**——保留判断多样性 = 降低系统性风险 + 留住个人传承
- **给出决策权的明确归属**——最终用哪个分身，是业务方决定，不是 AI 自己决定

#### 应答稿（现场复述版）

"这是个非常关键的问题，技术和业务都很重要。我直接给答案：**我们系统从一开始就是多分身架构，不是单一风格架构**。

**核心设计理念**：
- **不是'训练出一个综合 AI'**，然后把所有专家的判断平均掉
- **而是'养出多个 AI 分身'**，每个分身对齐一位专家的判断风格
- **业务使用时，由值班员或会商主持人决定'现在用哪个分身 / 分身们怎么用'**

**技术实现上**（讲到架构层，细节 NDA 后）：

- L4 控制器层的慢参数**不是一个'全局状态'**，而是**一组按'视角 ID'条件化的参数痕迹**
  - 张三的慢参数痕迹 → AI-张三分身
  - 李四的慢参数痕迹 → AI-李四分身
  - 试点负责人的慢参数痕迹 → AI-试点分身
- 使用时，系统**根据调用时指定的视角 ID**加载对应的干预向量，**前向推理中注入对应的判断倾向**
- 几位分身之间**参数是隔离的**——学张三不会冲掉李四

**三种典型使用模式**（业务侧自由选择）：

| 使用模式 | 场景 | 机制 |
|---|---|---|
| **单分身调用** | 某海区老张负责多年，值班员想要老张的判断视角 | 选 `@AI-张三`，单一风格输出 |
| **多分身联合会商** | 重大天气过程会商，想听多位专家视角 | 同时调用 3 位分身，每位独立给判断 + 理由，像一场虚拟会商 |
| **多分身加权综合** | 常规预报需要一个'总体建议' | 按预设权重融合（权重由业务副总 / 质量体系确定，可动态调整） |

**业务价值**：

1. **保留判断多样性**——这在气象 / 海洋这种**有极端长尾事件**的领域至关重要。系统性风险最大的来源就是'所有预报员都犯同一种错'——多分身天然对冲
2. **留住专家个人传承**——老张退休后 AI-张三分身还在，**他的判断风格以数字形式留在系统里**，年轻预报员可以从这个分身里学习、反思、对照
3. **降低 AI 被单一专家偏见污染的风险**——即使某位专家有系统性偏差，也只局限于他自己的分身，不污染其他分身
4. **权力归属清晰**——不会产生"谁是 AI 的官方老师"这种内部敏感问题

**决策权归属**（明确写入合作协议）：

- **哪些专家值得有分身**——中心质量体系与业务副总决定
- **分身间的权重 / 综合策略**——中心业务流程决定
- **某个分身停用 / 归档**——中心有权单方面决定
- **我们团队不涉入这些决策**——我们只提供多分身架构能力

**与杨柳博士研究方向的关联**：

这个机制在理论上叫做**"条件化学徒学习"**（Conditional Apprenticeship Learning）——学徒对多位师傅学习，但保留每位师傅的风格独立性，而不是强行融合。这也是杨柳博士近年持续研究的方向之一。

**业务上的一个典型操作场景**：

> 重大过程会商时，主持人说'我想听听老张和老李怎么看' → 系统并行调用 AI-张三 + AI-李四 + AI-会商主持人三个分身 → 每个分身给出**各自视角的判断 + 不确定性 + 理由链** → 主持人和其他与会者参考三个视角做最终决策 → 签发仍然是人类完成

这个模式我们在**工程结对场景**里已经验证——让一个团队的 3 位资深工程师各自的 AI 分身一起做代码 review，比单一 AI 的建议更有用也更有多样性。

#### 核心佐证点

- **多分身架构示意图**（可现场白板画）
- **工程结对场景的 3-分身联合 review 案例**（NDA 后展示）
- **分身间独立性的技术验证**——学一个不影响另一个的实测数据
- **分身归档 / 停用 API 列表**——证明业务方真的有决策权

#### 可能的追问与应对

| 追问 | 应对要点 |
|---|---|
| "多分身是不是就是多存几份参数？这不是简单的 'N 个 LoRA' 吗？" | 表面相似但机制不同。N 个 LoRA 是**离线训练出的静态适配器**，上线后不再变化；多分身是**每个都在持续通过学徒学习演化的动态控制器**。更重要的是，**多分身共享同一套契约层和共同语境**——它们不是'几个独立 AI'，而是'同一个 AI 的多种判断视角'，共享其他模块。 |
| "多分身会不会带来计算开销按 N 倍放大？" | **不会**。推理时只加载调用的分身（多分身联合调用除外）。慢参数体量很小（残差流干预向量规模远小于基座），N 份慢参数参数化的额外存储 / 计算可忽略。**多分身联合会商**模式因为确实要并行推理，开销会是 N 倍——这也是为什么联合会商是'重大天气过程按需开启'，不是日常默认模式。 |
| "一位专家退休或离职后，他的分身归谁？数据伦理怎么处理？" | **这是严肃的伦理问题**，我们合作协议里会包含：**（1）** 专家本人对自己分身享有知情权和停用权；**（2）** 专家可要求在离职时归档 / 删除分身；**（3）** 专家的分身在业务使用时带有明确署名。具体条款与中心人事 / 法务一起起草。 |
| "如果几位专家的判断在一个具体案例上**完全相反**，AI 的多分身综合判断会怎么处理？" | 系统**不做隐式折中**。在'加权综合'模式下，如果两个分身判断分歧度超过阈值，系统会**显式暴露分歧**——告诉值班员'张三分身建议发 3 级预警、李四分身建议不发，分歧原因大概在 X 因素上'。**由人类做最终判断**。这是 R7 / R14 的要求：保留对分歧的显式表达而不是抹平。 |

## C. 可追溯 / 合规

### Q8 · PPT 里说"AI 的内部记忆、判断倾向、权重变化都能追溯"——底层具体如何实现？存储和计算开销到中心能不能接受的量级？

#### 对方为什么会问

这是 deck 陷阱②的**技术落实问题**——"AI 日志只记录输入输出，不记录内部"是中心信息化 / 网安 / 质量体系处长一听就很敏感的命题。deck 宣称能解决，但具体**怎么实现**没讲——这一层不讲清楚，陷阱②就只是营销话。

同时对方会算账：**内部状态记录下来要多少磁盘？会不会拖慢业务？**——必须给出具体量级。

#### 应答策略

- **先讲追溯对象**：哪些内部状态被追溯
- **再讲数据结构**：类 Git 的不可变版本化
- **再讲存储算账**：给具体数字
- **最后讲性能开销**：写入 / 回退的延迟

#### 应答稿（现场复述版）

"这一层是我们和业界其他 AI 产品最大的差异之一，我讲得具体一点，因为这也是中心信息化 / 网安部门最关心的。

**先说追溯对象**——我们追溯的不是'AI 说了什么'（这个所有 AI 系统都能做），而是**'AI 内部表征随时间的全过程变化'**。具体分五类：

| 追溯对象 | 含义 | 频率 |
|---|---|---|
| **控制器慢参数版本** | L4 残差干预向量的长期状态，每次通过门控就产生新版本 | 天 / 周级 |
| **每次学徒交互记录** | 专家点评 → 候选更新 → 是否通过门控 → 最终是否进入慢参数 | 每次交互 |
| **每次前向推理的干预轨迹** | 本次推理用了哪些慢参数、哪些快参数、干预向量具体数值 | 每次推理 |
| **记忆系统事件** | 哪条记忆何时被写入、何时被召回、何时被淘汰 | 每次 |
| **契约 / 共同语境快照** | 多 AI 协同时的共同语境随时间演化的每一个版本 | 每次更新 |

**再说数据结构**——我们用的是一套**类 Git 的不可变版本化存储**：

- **每次状态变化 → 产生一个 commit**
- **commit 内容**：变化前哈希、变化后哈希、变化增量、触发源（哪次交互 / 哪条反馈 / 哪个评估信号）、时间戳、签名
- **commit 之间构成一个 DAG**（有向无环图），类似 Git 的提交历史
- **可以 checkout 到任意历史 commit**——这就是'回退'
- **所有 commit 不可修改**——类似区块链，一旦写入只能追加不能篡改
- **关键哈希落链可选**——如果中心要求，关键 commit 的哈希可以上**中心自有的审计链**（内部联盟链 / 区块链存证系统），做到外部可校验

**存储开销算账**（这是最关心的部分）——

我们在情感陪伴场景的实测数据（一个用户、6 个月在线学习）：

| 项目 | 典型体量 |
|---|---|
| **慢参数一次 commit** | 几百 KB 到 3 MB（增量存储，不是全量） |
| **快参数 / 干预向量一次推理记录** | 约 10–50 KB |
| **学徒交互记录 + 候选更新** | 单次约 100–500 KB |
| **记忆事件记录** | 单次几 KB |

**按试点 3 个月 / 一位专家 / 覆盖某海区日常值班 + 10 场典型天气过程估算**：

| 来源 | 体量 |
|---|---|
| 前向推理记录（预计每天几百次） | 每月 ~2–10 GB |
| 学徒交互 + 候选更新（10 条 / 周） | 每月 ~50 MB |
| 慢参数 commit（每周 1–2 次） | 每月 ~30 MB |
| 共同语境快照 | 每月 ~100 MB |
| 记忆事件 | 每月 ~500 MB |
| **合计** | **每月 ~3–12 GB，3 个月总量 ~10–40 GB** |

**对于中心 PB 级的气象数据基础设施，这是一个几乎可忽略的量级**。

**性能开销**（写入 / 读取延迟）——

| 操作 | 延迟 |
|---|---|
| 一次前向推理追加记录 | < 10 ms（异步写入，不影响主链路） |
| 一次慢参数 commit | < 500 ms |
| 回退到任意历史 commit | 秒级（按版本载入，不需要重建） |
| 全链路审计查询（'某时刻 AI 为什么这么判断'） | 秒到分钟级，取决于查询跨度 |

**和传统日志的根本区别**：

| 维度 | 传统日志 | 我们 |
|---|---|---|
| 追溯什么 | AI 的外部 I/O | AI 的**内部状态演化轨迹** |
| 因果关系 | 无法从日志反推'为什么' | 每个判断都可溯源到具体的参数版本 + 具体交互 |
| 回退 | 无此概念 | 一等公民 |
| 篡改防护 | 靠访问控制 | 哈希链 + 可选落审计链 |

**一个典型的追溯查询示例**：

> 质询：'2026 年 7 月 15 日 14:30，AI 对渤海湾台风 X 建议发布 3 级预警。请追溯这个判断的来源。'
>
> 系统输出：
> - 本次前向推理使用的基座版本：瀚海智语 vX.Y（sha256: …）
> - 使用的控制器慢参数版本：v237（2026-07-01 的 commit，触发源：张三 2026-06 第 4 周 #C 案例点评）
> - 本次推理激活的干预向量：…

这份证据卡**可导出、可签字、可作为对外披露依据**——正是陷阱②里说的'新范式'的工程落地。

#### 可能的追问与应对

| 追问 | 应对要点 |
|---|---|
| "每次推理都记录干预向量？那推理一多存储不就爆了？"  | 两级策略：**热存储**（近 30 天全量）+ **温 / 冷存储**（超过 30 天只保留**抽样 + 关键事件**）。热温冷策略由中心信息化规范决定，我们只提供机制。**关键事件永远保留**——包括所有对外发布的预报对应的推理轨迹、所有预警发布对应的推理轨迹等。 |
| "这和模型的训练 checkpoint 有什么区别？" | 大模型 checkpoint 是**几百 GB / 个**，保留几十个版本就 TB 级；我们的 commit 是**增量存储，每个 MB 级**。而且我们的 commit 保留**触发源和因果链**，checkpoint 不保留。 |
| "落审计链是不是意味着要搭区块链？中心现在没这个基础设施啊" | 落链是**可选增强**，不是必需。最基础配置只用哈希链本地存储 + 定期交叉签名 + 中心自有数据库，**不需要额外基础设施**。如果中心后续建了联盟链，我们只要加一个推送器就能把哈希上链。 |
| "整个追溯系统本身会不会被篡改？你们团队不能偷偷改吗？" | 几层机制：**（1）** 生产环境**我们没有远程访问权**（全私有化部署）；**（2）** 所有 commit 带数字签名（中心自主可控的 CA）；**（3）** 定期由中心将 commit 根哈希异地备份 / 上链；**（4）** 我们可以签承诺：**'任何篡改可被中心独立检出且构成违约'**。 |

### Q9 · "可回退"到底能回到什么粒度？一次交互能精确回退吗？整场会商能整体回退吗？回退以后系统行为是完全一致的吗？

#### 对方为什么会问

这是**信息化 / 质量体系负责人的"事故演练视角"提问**。他们脑子里有一个很具体的场景：

> **"某天 AI 发了一条明显错的预警，事后查是上周某次学徒交互把 AI 带偏了——我们能不能把'那次交互对 AI 的影响'干净地撤销掉，又不影响后面正确的学习？"**

这是他们判断"系统能不能进生产"的**红线问题**。含糊回答（"可以回退"）不够，必须讲清：**回退到哪个粒度、代价多少、回退后行为是否真的等同**。

#### 应答策略

- **先给粒度清单**——支持哪几级回退
- **讲清代价 / 时效**
- **讲清一个关键问题**：回退后是"完全恢复"还是"近似恢复"
- **把试点里的回退 SOP 勾出来**

#### 应答稿（现场复述版）

"这是个必须给明确答案的问题。我们支持**四个粒度**的回退：

| 粒度 | 回退什么 | 触发场景 | 代价 / 时效 |
|---|---|---|---|
| **L1 · 单次推理回退** | 本次推理的输出作废 | 某条具体输出有问题 | 毫秒级。但只是丢这一次输出，不影响后续学习 |
| **L2 · 学徒交互回退** | 撤销某一次学徒交互及其触发的候选更新 | 发现某次点评是错的 / 情绪化 / 被操纵 | 秒级。如果该交互已经进入慢参数 commit，回退同时回滚那个 commit |
| **L3 · 慢参数版本回退（主要回退机制）** | 慢参数 checkout 到某个历史版本 | 发现某段时间的学习让 AI 判断变差 | **秒级**。等价于选一个历史版本重新加载 |
| **L4 · 整体系统回退** | 整个系统（慢参数 + 记忆 + 契约 + 共同语境）回退到某个时间点的快照 | 重大事故演练 / 系统性问题 | 分钟级。需要一份时间点快照，通常定期自动生成 |

**关键问题：回退后系统行为是不是真的'完全一致'？**——分两种情况：

**情况 A · 严格等同**：
- 如果回退后**输入完全一样、环境完全一样**（比如在同一台机器上、同样的随机种子、同样的温度参数），**输出完全等同**
- 这是因为我们的前向推理是**确定性的**（可复现构建要求），没有任何隐式状态
- 这也是为什么传统数值模式可以'同样输入同样输出'——我们保留了这一点

**情况 B · 近似等同**：
- 如果回退之后又继续用了——后续的用户交互、RAG 召回结果可能不同——未来状态会发生**合理偏移**
- 这不是 bug 而是设计：**因为业务在变化，完全复刻过去没有意义**；关键是**回退点本身的判断能力是恢复的**

**一个具体回退 SOP 示例**（情感陪伴场景有多次演练）：

> 发现：2026-07-10 发现 AI 对某类用户诉求响应变差，怀疑是 2026-07-03 某次学徒交互带偏的
>
> **Step 1** 在追溯系统中查询：2026-07-03 该交互对应的 commit ID = v237
>
> **Step 2** 执行 `revert v237`：
>   - 慢参数回退到 v236
>   - 由该交互触发的记忆写入标记为 'reverted'（不物理删除，便于审计）
>   - 后续基于 v237 派生的 commit（v238 等）也一起回退
>
> **Step 3** 自动回归测试：在预留评估集上验证回退后指标是否恢复
>
> **Step 4** 出具一份**回退证据卡**：什么时间、什么原因、回退到哪个版本、验证指标——一份可签发的文档

这个流程**完整走下来不到 10 分钟**。

**边界承诺**：

- 所有回退操作**需要双人授权**——单人不能触发
- 回退操作**本身也被追溯**——`revert` 也是一个 commit，可以被再次 revert（回到 revert 之前的版本）
- 回退不会**删除历史数据**——只是把'当前状态'指向历史某个版本
- 回退**不影响瀚海智语基座**——基座从来就没动过

**对试点的承诺**：

- 3 个月试点中**我们主动安排至少 2 次回退演练**——请中心质量体系人员在场观测
- 演练内容：人为构造'AI 被学徒点评带偏'的情境，走完整回退 SOP
- **演练结果作为试点交付物之一**——写入评估报告

#### 核心佐证点

- **情感陪伴场景实际回退案例**（完整的回退 SOP 日志，NDA 后展示）
- **回退延迟实测数据**
- **回退前后评估指标对比图**（证明回退真的有效）
- **revert 操作自身的审计记录**

#### 可能的追问与应对

| 追问 | 应对要点 |
|---|---|
| "L3 回退能把上周所有正确的学习也退掉吗？我只想退一次错的，不想退掉别的正确的" | **可以做到**。因为 commit 之间的关系是 DAG 不是线性——每个 commit 记录'前置依赖'。如果 v237 和 v238 没有依赖关系（由不同交互触发、互相独立），可以单独 `revert v237` 保留 v238。类似 Git 的 cherry-pick 能力。**但如果依赖了（例如 v238 是基于 v237 的基础上增量），则需要同时 revert**——系统会显式提示这个依赖关系。 |
| "回退以后那条学徒交互本身还在数据库里吧？下次反思引擎跑的时候会不会又把它学回来？" | 不会。被 revert 的交互会被标记 `reverted=true + reason=...`，反思引擎有**硬规则**：**标记回退的交互不进入后续候选更新生成**。这个标记也是可追溯的。 |
| "要是需要法律证据怎么办？比如责任认定说'AI 在那个时刻的判断依据是什么'" | 这正是证据卡的价值。**任何历史时刻的 AI 状态都可以 replay**——回到那个时刻的慢参数版本、加载当时的输入、重现推理过程、输出当时的证据卡。这份证据卡带**时间戳 + 版本哈希 + 数字签名**，符合电子证据规范。 |
| "如果中心要求'某些学到的东西永远不能被回退'（比如法规要求的禁忌），怎么实现？" | 这是通过**契约层**而不是回退系统实现的——**契约层的硬约束凌驾于慢参数之上**，无论慢参数怎么变化，违反契约的判断会被拦截。所以'永不违反的禁忌'不是'不能回退的学习'，而是'永远生效的硬约束'。Q12 会讲这个机制。 |

## D. 多 AI 协同

### Q11 · "共同语境"的技术实现到底是什么？是一个全局状态机 / 全局字典吗？跨机构部署（国家中心 — 区域预报中心 — 地方）时怎么同步？CAP 怎么取舍？

#### 对方为什么会问

这是**懂分布式系统的架构师的专业提问**。deck 里的"共同语境"听起来很有道理，但任何做过分布式系统的人都知道——**全局共享状态 = CAP 取舍 + 延迟 + 一致性 bug**。如果答不上来具体架构，这个陷阱③的解决方案就是空话。

**海洋预报业务还有一个特殊性**：**跨机构部署**——国家中心、东海分局、南海分局、沿海省市级监测机构，网络互联有严格边界，强一致性基本不可能。

必须给出一个**在这种约束下仍能工作**的架构。

#### 应答策略

- **明确"共同语境不是全局状态机"**
- **讲清它的结构**：一组具名快照（slots），每个 slot 有明确 owner
- **讲清跨机构同步**：联邦式、最终一致性、契约守卫兜底
- **CAP 取舍**：选 AP，但用契约守卫让 C 风险可控

#### 应答稿（现场复述版）

"这个问题问得非常专业。我直接给答案：**共同语境不是一个全局状态机，也不是一个共享字典**——它是一组**具名的、类型化的、带明确所有权的快照**（slots），通过**发布-订阅**模式协同。这是我们从第一性原理推出来的，背后有一套严格的契约理论。

**核心数据结构**：

```
共同语境 = {
  slot_A:  { owner: module_X, version: v37, value: {...}, ts: T1, hash: ... },
  slot_B:  { owner: module_Y, version: v12, value: {...}, ts: T2, hash: ... },
  slot_C:  { owner: module_Z, version: v201, value: {...}, ts: T3, hash: ... },
  ...
}
```

**关键设计原则**（四条铁律，来自我们的契约规范）：

1. **每个 slot 有且只有一个 owner**——其他任何模块 / AI 只能读、不能直接写
2. **slot 值是不可变的**（frozen）——要更新 owner 必须发布新版本、老版本保留
3. **跨机构 / 跨模块通信只走 slot 读取**——**禁止直接调用对方接口**
4. **每个 slot 的 schema 集中定义在一份注册表**（`slot registry`）里，任何新 slot 必须先注册

**在海洋业务场景里的一些典型 slot**（示意）：

| slot 名 | owner | 含义 |
|---|---|---|
| `plan_intent` | 值班 AI | 当前值班意图（这段时间重点关注什么） |
| `commitment` | 签发者 | 对外已承诺的预报内容（Q12 会讲覆盖定义） |
| `open_loop` | 应急 AI | 尚未关闭的应急处置环节 |
| `user_model` | 主持人 AI | 当前用户 / 受众模型 |
| `execution_result` | 各执行 AI | 各自的执行结果快照 |
| `belief_assumption` | 预报 AI | 当前判断所基于的关键假设 |
| `relationship_state` | 协同 AI | 多方协同关系状态 |
| `boundary_consent` | 合规 AI | 当前业务边界与授权状态 |

每个 slot 有自己的 schema、owner、lifecycle——这个架构我们在 `DATA_CONTRACT.md` 里已经形式化定义好了。

**跨机构部署怎么办**——联邦式架构：

```
  国家中心
  ┌───────────────┐
  │ slot registry │ ← 定义全局 schema
  │ + 国家级 slots │ ← 国家级 AI 发布的 slot
  └──────┬────────┘
         │ 双向订阅 (标准 API)
    ┌────┴──────┬──────────┐
    │           │          │
  东海分局    南海分局    北海分局
  ┌─────┐   ┌─────┐    ┌─────┐
  │区域│   │区域│    │区域│  ← 各自管理本地 slot
  │slots│   │slots│    │slots│
  └──┬──┘   └──┬──┘    └──┬──┘
     │         │           │
  省市县级 AI 订阅
```

**关键机制**：

- **国家 slot registry** 是所有 schema 的权威来源（类似 DNS 根）
- **各级机构有自己的本地 slot 空间**——管辖区域内的 slot 本地拥有
- **跨级订阅走标准 API**——类似目前中心和区域中心之间的数据交换通道
- **数据交换遵循机构边界规则**——哪些 slot 可跨级、哪些不可，由业务 / 安全规范定

**CAP 取舍**——我们明确选 **AP**（可用性 + 分区容忍性），**弱 C**（最终一致性）。理由：

- **强一致性代价太高**：海洋业务跨省市跨海域，网络延迟、链路中断都是常态，强一致性会导致业务阻塞
- **强一致性也不现实**：中心 - 区域 - 地方之间现有的数据交换本来就是异步的，AI 协同层也应该一致
- **但弱 C 不能放任**——**用契约守卫兜底**：

**契约守卫**（contract guard）是关键的补偿机制——它在**每个 AI 消费 slot 时做运行时校验**：

| 守卫项 | 检查内容 | 触发时行为 |
|---|---|---|
| **版本陈旧检测** | slot version 是否超过容忍窗口 | 标记为 '过期语境'，降级 / 暂停相关判断 |
| **跨级一致性检测** | 本地快照与上游快照是否存在矛盾 | 触发人工仲裁或自动回滚到上游版本 |
| **schema 兼容检测** | slot value 是否匹配注册的 schema | 拒绝消费，报警 |
| **义务完成性检测** | 已承诺的 slot 是否按时更新 | 触发预警（'某区域 AI 未按时发布应急态势'） |

这个守卫层让我们**用'AP + 强约束的契约层'替代了传统的强一致性**——不依赖底层分布式存储的强一致（那代价太高），而是**让每次消费都显式验证语境的健康度**。

**最关键的设计选择**：**宁可暴露不一致，也不隐式掩盖**。如果守卫检测到某片区域 AI 的语境落后超过阈值，**系统会显式报警、暂停跨域协同、提示人工介入**——而不是静默地让每个 AI 按自己的本地状态自说自话。

**这正是 deck 陷阱③举的那个例子的解决方案**：
> '台风 X 三级预警，国家中心 AI 输出"覆盖 = 福建全省"'——在我们的契约体系里，'覆盖' 这个词对应一个 slot（`commitment.coverage`），schema 规定 **'覆盖'必须展开到具体行政级别**（地市级 / 市县级 / 乡镇级）。国家中心 AI 发出 slot 时就必须写清楚到哪一级；区域中心 AI 订阅时拿到的是同一份展开版；**语义自动对齐、不可能模糊**。漏发警报的事不会发生。

#### 核心佐证点

- **DATA_CONTRACT.md**（契约 + slot 注册表形式化定义，我们的内部规范，NDA 后可递交）
- **两个领域中的 slot 注册表样本**——证明这套框架在实际产品里已经运行
- **跨机构联邦部署拓扑图**（现场白板可画）
- **契约守卫实际触发日志**（证明兜底机制在真正工作）

#### 可能的追问与应对

| 追问 | 应对要点 |
|---|---|
| "你们的 slot 是不是就是 Redis / etcd？有什么必要造个新词？" | 不是。Redis / etcd 是**存储介质**，我们的 slot 是**具有类型、版本、所有权、lifecycle、schema 校验的语义单元**——可以存在 Redis 里，也可以存在 PostgreSQL 里，**存储选什么不重要，重要的是上面那一层结构**。你可以理解为：**slot 是一等的业务语义对象，存储只是实现细节**。 |
| "你们这个联邦架构和气象局的 CIMISS、CMADaaS 数据系统有什么关系？会不会冲突？" | 不冲突，甚至**可以叠在它们之上**。CIMISS / CMADaaS 是**物理数据（观测、实况、模式）的分发平台**；我们的共同语境是**AI 协同状态的分发平台**——两个不同层级的共享基础设施。可能会有一些 slot 引用 CMADaaS 的数据作为 value，但 slot 框架本身独立。 |
| "联邦 slot 跨机构传输，会不会泄漏敏感内容？" | 每个 slot 的 schema 里标注**跨域可见性**（'本地 only / 区域可见 / 全国可见 / 公开'）——这是 DATA_CONTRACT 规定的必填字段。跨机构同步时，**只同步该级别允许跨域的 slot**，其他自动过滤。这个机制和涉密数据交换的 '分级分类'是同构的。 |
| "守卫层每次都要校验，性能会不会是瓶颈？" | 守卫校验的开销是**毫秒级**（hash 验证 + schema 匹配 + 版本比较），相对于 AI 推理（几百毫秒到秒级）是可忽略的。在两个落地场景的高并发情境下做过压测，守卫层不是瓶颈。 |
| "万一某个 slot 的 owner 挂了，全链路不就卡住了？" | 每个 slot 在 schema 里定义**容忍窗口**和**降级策略**——owner 断联超过阈值，消费者进入**降级模式**（用最后已知 value + 明确标注'过期'）；同时触发报警。业务链路**不会因为一个 owner 挂了而整体阻塞**。 |

### Q12 · "覆盖""承诺""完成"这些词的语义形式化——Schema 到底谁来定义？是不是要建一个巨大的本体论（Ontology）？海洋业务词汇浩瀚，你们能搞得定吗？

#### 对方为什么会问

这是**老预报员 + 业务流程负责人联合提问**。deck 里的台风覆盖漏发案例很生动，但懂业务的人会问：

- 光一个"覆盖"就有几十种含义（空间覆盖 / 时间覆盖 / 对象覆盖 / 责任覆盖…）
- 光一个"承诺"——预报承诺、警报承诺、应急承诺、服务承诺、时限承诺，每种都有自己的规则
- 海洋预报涉及数千个专业术语——完整建模是"人生没那么长的事"

**如果你们说要建一个大一统本体论，这个方案就死了**——没人信能建成。

同时会担心：**Schema 谁定义？我们业务处还是你们技术团队？定义错了谁负责？**

#### 应答策略

- **明确反对大一统本体论路线**——这路早已被学术界证伪
- **讲清我们的选择**：**小而精 + 业务驱动 + 按需生长**
- **Schema 定义责任划清**：中心业务方定义语义、我们提供工具
- **给一个具体的"覆盖"形式化示例**
- **说明如何应对新词 / 新语义的涌现**

#### 应答稿（现场复述版）

"这个问题问到了所有'语义工程'项目的坟场——**试图一次性建完整本体论的项目，过去 30 年几乎都失败了**。我们**明确不走这条路**。

**我们的路线叫 '按关键协同点形式化'**（formalize at critical handoff points），不搞大一统本体。核心原则：

**原则 1 · 只形式化真正引起多 AI / 多机构协同冲突的词**

海洋业务有几千个专业术语——绝大多数是**单个 AI 内部使用、不跨边界**，不需要形式化。只有在**跨 AI / 跨机构协同时才有歧义风险**的词，才值得形式化。

根据我们对中心业务的初步观察（deck 里提到的陷阱③的例子），真正会引起跨边界歧义的词大概在 **20–50 个量级**——比如：

| 词 | 歧义来源 | 形式化重点 |
|---|---|---|
| **覆盖** | 空间粒度、对象层级、时间窗口都含糊 | 要求展开到：行政级别 + 时间窗 + 对象集合 |
| **承诺 / 已承诺** | 承诺给谁、承诺什么、时限 | owner + audience + content + deadline |
| **完成** | 做了就算完成？还是有人验收才算？ | owner + accepted_by + evidence |
| **生效 / 撤销** | 撤销之后是否追溯？追溯到哪 | effective_range + retroactive_flag |
| **超界 / 越权** | 哪种类型的越界（业务 / 空间 / 时间） | boundary_type + trigger_rule |
| **关注 / 重点** | 到什么关注级别、多久 | priority_level + window |

**这是一个'最小可工作词表'**，不是一个百万级的 ontology。

**原则 2 · Schema 定义责任的明确划分**

- **语义 + 业务规则**——由**中心业务方定义**。我们不懂海洋，也不应该代替中心决定'覆盖在渤海湾业务里应当包含到哪一级'
- **形式化框架 + 工具**——由**我们提供**。包括 schema 格式、校验工具、注册流程、变更管理
- **联合定义工作坊**——在试点启动阶段，我们和中心业务处联合办 2–3 场工作坊，挑出这 20–50 个关键词，**一起**写成形式化 schema
- **最终责任**——schema 一旦写入注册表就成为合约，**变更需要双方授权**。业务方有单方面撤回 schema 的权力，我们没有

这个分工和中心现有的"业务规范定义 + IT 实现"是同构的——我们就是 IT 实现方。

**具体来看"覆盖"这个词怎么形式化**（示例）：

```yaml
# 注册到 slot_registry 的 "coverage" schema
slot_name: commitment.coverage
owner: signing_authority_ai  # owner 必须单一
schema_type: structured

value_schema:
  spatial:                    # 空间覆盖必须展开
    granularity: enum [省 / 地市 / 市县 / 乡镇 / 格点]  # 必填
    entities: list[string]    # 具体的行政实体名 + 统一编码
  temporal:                   # 时间覆盖
    start_ts: datetime
    end_ts: datetime
    time_zone: string
  object:                     # 覆盖对象
    hazard_types: list[enum]  # 台风 / 风暴潮 / 涌浪 / 海冰...
    severity_range: enum
  audience:                   # 承诺给谁
    type: enum [内部 / 公众 / 应急系统 / 兄弟单位]
    recipients: list[string]

invariants:                   # 硬约束
  - spatial.granularity 必须填到"市县"级以下
  - entities 非空
  - end_ts > start_ts
  - hazard_types 与 object.severity_range 匹配对应表

lifecycle:
  mutable: false              # 一旦承诺不可隐式修改
  updates: "only via a new commit referencing previous"
```

**陷阱③的场景在这个 schema 下根本不会发生**——因为：

- "覆盖 = 福建全省" 这种字段**在 schema 校验时就被拒绝**（granularity 没填到市县）
- 国家中心 AI 必须显式写清 `entities = [福州市, 厦门市, 泉州市, ...]` 完整列表
- 区域中心订阅时拿到的就是这个完整列表——**语义自动一致**

**原则 3 · Schema 是'活的'——可以演化，按业务需要生长**

- 发现新的协同歧义点 → 走变更流程注册新 slot / 新 schema
- Schema 版本化管理（类似 Protocol Buffer 的向后兼容）
- **消费者容忍旧版本**（通过版本兼容规则），**发布者按新版本产出**——逐步迁移

**原则 4 · 起步不必完美——3 个月试点期足够验证核心词表**

- 试点前联合工作坊，**圈定 10–15 个最关键词**（不是 50 个）
- 3 个月试点里逐步补充
- 试点结束时形成一份 **'海洋协同语境基线 schema'**——后续可沿用到其他合作场景

**和传统本体论工程的差别**：

| 维度 | 传统本体论（失败路线） | 我们 |
|---|---|---|
| 目标 | 全领域语义建模 | 只管跨边界歧义词 |
| 词表规模 | 数万 | 数十 |
| 工程周期 | 多年 | 几周到几个月 |
| 维护方 | 专职本体工程师 | 业务方 + 我们联合 |
| 何时扩展 | 预先完整建模 | 按业务需要生长 |

所以回答'你们搞得定吗'——**我们不搞那种大一统的工程，我们搞的是小而精的工程，3 个月试点期内完全可以覆盖关键场景**。

#### 核心佐证点

- **两个领域的实际 slot 注册表**（情感场景 / 编程场景）——每个领域大概 30-40 个 slot，证明这个规模在实际产品里够用
- **联合工作坊 SOP 文档**（NDA 后可递交）——证明这是一个成熟方法论不是临时起意
- **Schema 变更管理工具**——Demo 可演示

#### 可能的追问与应对

| 追问 | 应对要点 |
|---|---|
| "业务专家不会写 yaml / schema，你们说让我们业务方定义，这不现实" | **业务方定义语义，我们负责把它翻译成 schema**——联合工作坊上专家用业务语言讲清'覆盖在这个场景里必须包含 X / Y / Z'，我们现场把它写成 schema 给专家看、专家确认。专家不碰代码。 |
| "这个'共同语境'是不是就是一个业务规则引擎（BRMS）？" | 有一定相似——都是'把业务规则明确化'。但 BRMS 的目标是'基于规则决策'，我们的目标是'跨 AI 协同时语义对齐'。我们的 schema 不描述决策规则，只描述**共享语境的最小必要展开度**——具体怎么判断、怎么决策仍由每个 AI 自己完成。两者是**正交的**，未来也可以把现有 BRMS 作为我们契约层的一个子模块。 |
| "新词出现太频繁怎么办？每来一个词就开一次工作坊？" | 不会那么频繁。在情感和编程两个场景里，slot 注册表**前 2 个月快速增长，之后趋稳定**——后续新增 slot 每月 1-2 个。海洋业务是成熟领域，术语在业务规范里已经相对稳定，预计增长更慢。 |
| "国标 / 行业标准有没有对接？比如 QX 系列气象标准、SL 系列水利标准" | 有相关 slot 时**优先对接国标**作为其 schema 的 `reference_standard` 字段。如果中心有现行的内部业务规范（例如海洋预报业务规范），schema 直接引用条款编号，保证一致性。这是联合工作坊的输入之一。 |

## E. 场景可迁移性

### Q13 · 说实话，情感陪伴、编程协作和海洋预报——这三件事的技术难度和业务特性差得太远了。你们在前两个领域成熟，不等于能搞定国家级气象业务。你们凭什么说"同一个内核"？

#### 对方为什么会问

这是**一把手 / 业务副总带着健康怀疑的核心质疑**。PPT 里"同一个内核已在两个领域跑通"是关键论据，但业务方眼里：

- **情感陪伴**：2C、没有对错、容错率高、失败不致命
- **编程协作**：工程师用、有代码测试反馈、错了 Ctrl+Z 就行
- **海洋预报**：国家级业务、上亿人生命财产、一条漏报就是国家级事故

**这三者的"工程严肃度"不在一个量级**。任何夸大"平移"都会被当场打脸。如果回答得像在推销、像在避重就轻，**一把手会直接判你不严肃**。

#### 应答策略

- **第一时间正面承认差距**——不要试图抹平
- **把"任务层差距"和"运行时层同构"清晰拉开**——这是核心论点
- **用具体共有特征说明为什么运行时层可平移**
- **举从情感到编程的跨度作为已验证的'大跨度迁移'前例**
- **在试点切入点上主动保守**——不承诺直接做台风预报

#### 应答稿（现场复述版）

"我非常同意这个质疑的分量，先正面承认：

**任务层上**，这三件事差别确实巨大：
- 情感陪伴：容错高、无客观对错、失败可复盘可重试
- 编程协作：有代码测试做客观反馈、错误成本有限
- 海洋预报：国家级业务、失败的代价远远超出前两者——**一条漏报可以是国家级事故**

**我们没有、也不敢**声称'做过情感陪伴就能做预报'。我们做过情感陪伴和编程结对 → **完全不等于**我们做过国家级气象业务。

**但我们说的'同一个内核'，指的不是任务能力，是'运行时能力'**——这两层差别巨大，我展开：

**任务层 vs 运行时层的区分**：

| 维度 | 任务层（每个场景都不同） | 运行时层（跨场景共通） |
|---|---|---|
| 承担者 | 基座大模型 + 业务知识 + 编排 | 我们这一层 |
| 海洋场景 | 瀚海智语 + 妈祖 + 业务规范 | 我们 |
| 情感场景 | 基座 + 用户画像 + 对话策略 | 我们 |
| 编程场景 | Code LLM + 代码库索引 + IDE 集成 | 我们 |
| 例子 | '给出该海区未来 24h 浪高预警' | '这个判断能不能追溯？能不能回退？是否和上周的判断一致？' |

**三个场景在任务层上差异巨大，但在运行时层上高度同构**——都需要：

- **长期关系的持续**（不是一次性问答）
- **少量高价值交互中的学习**（不是海量标注）
- **判断风格的个性化**（情感偏好 / 代码风格 / 预报保守度）
- **多 Agent / 多系统协同的语境共享**（情感场景有支持 AI + 危机 AI + 关系 AI / 编程场景有结对 + review + 调试多 agent / 海洋场景有跨中心跨系统 AI）
- **可追溯 / 可回退 / 可审计**（情感陪伴在未来也要面对合规 / 编程 review 需要溯源 / 海洋最严格）

**我们的内核专门解决的是运行时层**——**任务层我们碰都不碰**。也就是 deck 里那张四层图的意思：
- 基座（妈祖 + 瀚海智语）——中心的
- 编排和应用层——中心的
- 运行时层——我们这一层
- **任务层始终是中心的，我们不越界**

**一个更贴切的类比**：

就像 **JVM**——JVM 在电商、银行、电信、政务等完全不同的业务里都能用，不是因为 JVM 懂这些业务，而是因为 **JVM 解决的是'Java 程序跑得稳'这一类与业务无关的基础问题**。我们就是'AI 程序跑得久跑得稳'那一层。

**从情感到编程这一步，本身就是'跨度巨大'的验证**：

- 情感陪伴和编程结对在任务层上几乎没有共同点
  - 对象不同（情感是人本身 / 编程是代码）
  - 反馈不同（情感是情绪状态 / 编程是编译运行）
  - 成功标准不同（情感是长期关系 / 编程是代码质量）
- 但**同一个内核只改适配层就搞定了两个场景**——这就是'内核的领域无关性'的工程证明

第三步从编程到海洋，确实是更大一步——**尤其是合规、严肃度、跨机构协同这几块**。但我们的内核**原本就是按'工业级、合规友好、跨机构协同'的约束设计的**——情感和编程场景某种意义上是'更简单'的子集。

**所以我们的态度**：

- **运行时层**：我们非常自信——内核在这一层经过两代场景锤炼
- **任务层**：我们非常谦逊——海洋预报的任务层我们不懂、也不应该越界

**这也是为什么我们在 5 节的试点建议里故意选'辅助型切片'而不是'预报自动化'**：

| 建议试点 | 本质任务 | 选择理由 |
|---|---|---|
| A. 区域智能预报系统 | 判断一致性辅助 | 运行时能力直接可验证 |
| B. 预报图自动解读 | 长期稳定辅助工具 | 不涉入签发、零风险 |
| C. 应急会商智能体 | 跨轮记忆 + 审计 | 运行时陷阱③ 直接可验证 |

**三选一都是'人工做主、AI 辅助'的模式**——我们的价值通过这些辅助工作展现，**不会在试点阶段就直接替代任何预报判断**。这也是 5 条承诺里第一条'不动对方现有系统'的意思。

**最后一句**：海洋预报是我们的第三步，不是第一步——**如果第三步是直接替预报员签字，我们不会选**。我们选的是'在已有业务里补一层长期记忆 + 追溯 + 协同'，这一层**恰恰是我们的专长，也是中心目前最缺的那层**。

#### 核心佐证点

- **两个领域共用内核的架构对比图**——逐模块显示 'identical / adapter / specialized' 的色标
- **从情感到编程迁移时的代码改动统计**——改了几行、哪几层、花了多少工期
- **适配海洋场景的改动预估表**——哪一层需要适配、工作量估计
- **试点建议选项的风险分析表**

### Q14 · 海洋预报是强物理约束（数值模式）+ 强客观评估（TS 评分、漏报率、GS）的业务。你们的"判断风格""气质"听起来主观——怎么和这些客观指标对齐？别搞成"AI 学得像专家了但预报反而更差"

#### 对方为什么会问

这是**首席预报员 / 质量考核负责人的硬命题**——他们的考核 KPI 就是 TS、漏报率、预警准确率、发布时效。任何一个'AI 辅助'方案，如果不能直接对齐到这些指标，就进不了业务。

而且他们会担心：**"AI 学了老专家的'保守直觉'，结果在不该保守的时候也保守，漏报率反而升了"**——这是非常现实的风险。

这个问题答不好，deck 里的核心价值承诺会被当成"华而不实"。

#### 应答策略

- **首先承认 TS、漏报率是业务的'硬考核'，不能被'判断风格'绕开**
- **讲清'判断风格内化'和'客观指标提升'之间的正确关系**——辅助、不是替代
- **给出可直接量化的评估方法**——成对对比
- **给出硬约束机制**——契约层拦截违反物理规则的输出
- **直接承诺：如果试点结果 AI 辅助组 TS 弱于纯人工组，试点算失败**

#### 应答稿（现场复述版）

"这个问题是海洋业务最硬的考核线，我直接回答，不绕。

**第一层承认**：TS、漏报率、GS、预警准确率、时效等是**业务的客观硬指标**，不能被'判断风格'或'气质'这种主观概念绕开。如果我们的系统让这些指标下降了，再漂亮的'学像了老专家'都是失败。

**第二层澄清**：'学专家判断风格'和'客观指标'之间的关系不是对立的，是**辅助关系**：

| 概念 | 含义 | 业务指标 |
|---|---|---|
| 预报准确度 | 数值给出的量 + 专家订正后的最终量 vs 实况 | TS / RMSE / GS |
| 漏报 / 误报率 | 业务规则的阈值判断是否及时准确 | 漏报 / 误报率 |
| **判断风格** | 在**存在多个合理选项**时（数值模式不确定 / 多源分歧 / 罕见情境）**倾向**于哪个 | 不直接对应 |

**核心事实**：TS、漏报率等指标**主要取决于'数值模式 + 专家订正 + 业务流程'**——我们不改数值模式，所以**我们对这些指标的直接贡献是有限的**。

**我们的价值在哪儿**：

- **在'存在判断分歧'的关键决策点上，帮助人类专家做出'更像老专家的决策'**
- **把判断过程和依据结构化，减少决策时间和沟通成本**
- **多 AI 协同语境保证不漏发警报**（这是陷阱③的直接贡献）
- **长期来看，让老专家的判断风格不随退休流失**——这对 10 年后的 TS 更关键

**所以正确的对齐方式**是：

**AI 辅助后整体业务的 TS、漏报率、时效指标 ≥ 纯人工基线**

**不要求**：
- AI 独立 TS ≥ 人工（我们不出独立预报）
- AI 提升 TS 绝对值（我们的价值在流程、在协同、在长期）

**第三层落到评估方法**——**成对对比**（paired comparison）：

```
对同一组历史天气过程，做两组并行：

  组 A：纯人工会商 / 签发          组 B：AI 辅助 + 人工会商 / 签发
    ↓                              ↓
    TS_A                           TS_B
    漏报率_A                        漏报率_B
    决策时间_A                      决策时间_B
    专家主观满意度_A                 专家主观满意度_B

硬指标：TS_B ≥ TS_A，漏报率_B ≤ 漏报率_A（不劣化）
附加指标：决策时间、满意度的改善
```

这个评估方式的好处：**两组基准相同，差异只是 AI 辅助的有无**——可以严格归因。

**具体到 3 个月试点的指标清单**（试点启动前与中心质量体系敲定）：

**硬指标**（绝对要求）：
- TS / RMSE / GS 在试点关注的过程上**不劣化**
- 漏报 / 误报率**不劣化**
- 发布时效**不劣化**

**软指标**（价值体现）：
- 决策时间减少 / 会商效率改善
- 跨 AI / 跨机构协同中字段语义一致率（对应陷阱③）
- 审计回溯时效（给定一条 AI 建议，完整溯源需要的时间）
- 老专家每周 2 小时的投入产出比（内化速度）

**硬约束机制**（防止'学风格学飘了'）：

契约层的**业务规则守卫**是一道硬约束——**任何违反已知物理规律或业务规则的 AI 建议都被系统自动拦截**，不允许进入签发流程。举例：

| 业务规则 | 契约守卫 |
|---|---|
| 预警级别与最大风浪量的对应表 | AI 建议的级别必须满足该表 |
| 预警范围必须覆盖某阈值以上的风险区 | 范围覆盖不足直接拦截 |
| 时效的最晚发布时限 | 超时自动标记 |
| 物理量间的必要一致性（风速 vs 浪高大致关系） | 不一致强制人工复核 |

这些规则由中心业务处 + 预报员在试点启动前定义，写入契约注册表，**系统硬性执行**——**AI 无论'学'出了什么判断风格，碰到这些硬约束自动触顶**。

**最硬的承诺**：

> **如果 3 个月试点结束时，在中心指定的 holdout 天气过程集合上，'AI 辅助 + 人工' 组的 TS 弱于或等于'纯人工' 组（即辅助没产生正向价值），试点算失败。我们不收尾款、不继续合作，交付所有数据和文档退出。**

这条写入合作协议。

#### 核心佐证点

- **两个领域的 A/B 对比实测报告**（情感：满意度对比 / 编程：代码质量对比）——证明我们在'辅助后整体不劣化且有改善'这点上有实操能力
- **契约守卫拦截日志**（证明硬约束机制确实在工作）
- **成对对比评估方案**（NDA 后递草稿）——由中心质量体系签字后生效
- **失败条款写入协议的草稿**

#### 可能的追问与应对

| 追问 | 应对要点 |
|---|---|
| "TS 评分的对象是什么？针对台风轨迹、风暴潮峰值、浪高场，还是警报发布？你们怎么选？" | **我们不选**。**评估对象由中心质量体系定**——中心既有的业务考核体系对每种产品都有标准 TS 定义，我们只对接、不重新定义。中心说要对什么评估，我们就对什么做 A/B。 |

| "AI 学了老专家的保守直觉，在新型气候下保守错了怎么办？" | 两层机制：**（1）** Q6 的门控——慢参数更新只在跨案例稳定时才进入；**（2）** Q14 讲的契约层硬约束——违反已知物理规则的保守性自动拦截；**（3）** 如果是纯业务判断差异——AI 只是 one of inputs，**最终决策由人类做**。 |

## F. 国产化 / 工程

### Q15 · 你们在通义千问开源权重上做的验证——和瀚海智语不一样吧？接入瀚海智语有多少工作量？昇腾 / 寒武纪算力上跑过吗？基座升级或切换时风险多大？

#### 对方为什么会问

这是**国产化与运维负责人的硬命题**。中心的国产化是国家战略硬指标，合作方声称"支持国产化"的话**每一个字都要可验证**。

对方知道的行业现实：
- 通义千问是稠密模型、基于标准 Transformer
- 瀚海智语是中心自研、可能架构和 tokenizer 都有定制
- 昇腾 CANN、寒武纪 MLU、海光 DCU 对 PyTorch 支持度各不相同，**算子移植是一摞坑**

**"都能跑"很容易说，但落到具体版本号、具体算子、具体性能**——行家会立刻知道是真懂还是装懂。

#### 应答策略

- **不装懂——承认没跑过瀚海智语和具体国产卡**
- **但讲清架构上为什么风险可控**
- **给出具体的适配工作流程和工期预估**
- **切换风险：分三类场景分别讲**
- **用一个硬承诺兜底：试点启动前做硬件兼容性评估，不能过就不开工**

#### 应答稿（现场复述版）

"我诚实回答。

**事实披露**：

- 我们**做过**通义千问（Qwen 开源权重）、DeepSeek、Llama 系 的验证——完整证据包在
- 我们**没做过**瀚海智语的适配——中心自研、非开源，我们没接触过
- 我们**没在昇腾 / 寒武纪 / 海光上跑过**——两个已落地领域场景都是 NVIDIA 基础上验证

**但架构上为什么风险可控**——三个关键事实：

**事实 1 · 我们这一层几乎没有平台相关代码**

我们的核心计算是：**PyTorch 前向钩子（hook）+ 残差流上的向量加法与标准矩阵乘法**。**没有自定义 CUDA 算子、没有 Triton 内核、没有任何架构相关的 kernel**。

这对于国产加速卡的意义是——只要目标平台支持**标准 PyTorch 2.x 前向 + 标准 hook API**，我们就能跑：

| 加速卡 | PyTorch 支持 | 我们的兼容性预期 |
|---|---|---|
| NVIDIA | 原生 | 已验证 |
| **昇腾**（CANN + torch_npu） | 官方适配 | 无自定义算子，**高度兼容预期** |
| **寒武纪**（cambricon-pytorch） | 官方适配 | 同上 |
| **海光 DCU**（DTK + torch_hip） | 官方适配 | 同上 |

**事实 2 · 瀚海智语的接入点仅涉及一类结构上的钩子**

不管瀚海智语内部是不是 MoE、是不是定制注意力——只要是基于 Transformer 家族的，残差流就客观存在，我们的干预钩子可以挂在残差流的标准节点上。

**主要适配工作**有三项：

1. **找到挂钩点**：瀚海智语每层残差流的访问点（需要中心提供必要的 API / 源码线索）
2. **Tokenizer / Embedding 维度对齐**：我们维护的慢参数维度需要匹配瀚海智语的 hidden_size
3. **前向回归验证**：确保挂钩后瀚海智语的原生输出**字节级不变**（bypass_mode），保证不掉点

预计工作量：**2–4 周双方工程师对接**。

**事实 3 · 基座切换时的风险分三级**

| 切换场景 | 风险 | 适配工作 |
|---|---|---|
| **同一模型版本升级**（瀚海智语 v1.0 → v1.1，结构基本不变）| 低 | 仅回归测试，慢参数可直接沿用 |
| **主版本升级**（架构微调、embedding 维度可能调整）| 中 | 接入点 + 维度重对齐，慢参数**映射迁移**或**快速重建** |
| **切换到异构基座**（如瀚海智语 → DeepSeek / 通义）| 高 | 适配层完全重跑；慢参数**不可直接迁移**，但**学到的专家风格可通过学徒交互重新积累**（因为中心保留的学徒交互记录是基座无关的） |

**关键设计**：我们的**学徒交互记录是结构化的、不绑定基座**——这意味着即使基座整体换掉，中心**积累的专家交互数据不丢失**，可以在新基座上重建判断倾向，这个重建不需要老专家再花一次 100 次点评，因为**历史点评可以作为重放材料**。

**对试点的硬承诺**：

1. **合作启动前 2 周**，由我们技术团队 + 中心硬件平台团队联合做**硬件兼容性评估**——在中心的国产加速卡环境上跑一个**端到端最小验证包**
2. 如果评估失败（例如某个算子不支持、性能不达标），**评估报告如实给中心，合作不启动**
3. **试点期间的所有运行必须在中心指定的国产硬件上**——不在 NVIDIA 上做开发绕过国产化要求
4. 每次基座变更时，**我们提供完整的适配报告 + 回归评估**
5. 我们**不锁定中心的基座选择**——未来中心切换基座是你们的权利，我们按上表工作量适配（工作量按合理工期计费，不附加锁定条款）

**关于私有化部署的资源开销**（顺带说一下，这个也是国产化要关心的）：

- 我们这一层**不承载基座推理**——主推理在中心现有的基座部署上，我们只在前向时插入 hook
- 额外 GPU 需求：**每个推理实例 +5–15% 显存 / 计算开销**（干预向量 + 慢参数缓存）
- 额外 CPU / 内存：独立服务的 2-4 路 CPU + 64-128 GB 内存
- 额外存储：参照 Q8 的算账

也就是：**我们的工程脚印远小于基座部署本身**。

**具体到产品形态的工程化成果**：

- 两个领域的交付级软件模块（不是 demo、不是 PPT）
- 完整的**契约注册表 + 模块边界 + 快照 SSOT 隔离**——这是我们内部代码库的硬规范（`docs/DATA_CONTRACT.md`, `ssot-module-boundaries.mdc` 等）
- **证据体系按论文级工程标准打包**（主张登记 / 对照组重复实验 / 盲评材料包 / 全链路溯源）
- **CI / 代码规范 / 错误不能静默吞掉**——我们的内部规则甚至禁止了 try/except + pass 这种'看起来容错'的写法

这些内部规范本身就是**严肃工程的侧面证明**——很多大厂团队也做不到。

**三、团队现状**：

具体人数我们倾向于**在 NDA 后披露精确数字**，但可以说的是：

- 核心团队**完整覆盖**了：首席科学家、算法工程师、后端工程师、基础设施、前端、产品、合规
- 核心成员**大部分不是短期合伙人**——是共同投入多年的稳定团队
- 为了三年内支撑多个场景落地，**正在有节奏地扩张**，但扩张速度**以代码规范吸纳能力为上限**——不盲目扩招

**四、国家级业务的应对方案**——三条策略：

**策略 1 · 我们不独自扛业务运营**（立场清晰）

我们的定位是**运行时底座提供方**——类似数据库、类似中间件供应商。日常业务运营、值班、应急响应、用户支持——**由中心现有的业务团队 + IT 团队主导**，我们提供**技术支持 + 定期现场工程 + 关键升级**。

这就是 deck 四层图里'基底、编排、应用都在中心侧'的实际含义——**我们只对中间那一层负责**。

**策略 2 · 合作早期阶段 = 中心技术骨干参与共建**

如果进入深度合作，我们强烈建议中心**派 2–4 名技术骨干加入项目**——不是作为'客户方'，而是作为**联合开发者 / 未来的内部专家**。

这样做有三个好处：
- 中心自己人掌握系统运行原理，降低长期供应商依赖
- 中心可以做**自主二次开发和扩展**，不被我们绑住
- 合作结束或任何特殊情况，**中心仍然能自主运维**

这个模式在**开源基础设施领域是常见的**（类似 PostgreSQL 国产化发行版的工作模式）——不是我们新发明的。

**策略 3 · 可选：与中心认可的系统集成商联合交付**

如果中心规模扩张或响应要求超出我们规模，可以**与中心既有的长期技术合作商（大型国产 AI 或系统集成商）结盟**——他们承接面向业务的实施、支持、运维，我们专注于内核与关键升级。

这个模式**保留我们的技术原创性不被侵蚀**，同时**借助成熟集成商的交付体量**支撑国家级业务规模。

**五、长期抗风险机制**——就算我们公司明天倒闭，中心也不被卡住：

1. **二进制 + API 交付**——中心部署后**有完整运行态**，不依赖我们在线运营
2. **关键源码第三方托管 + 触发释放**——触发条件在合同里写清楚（例如：我方停止提供更新超过 6 个月、公司解散、关键人员流失等），**触发即释放源码给中心**，中心自主维护
3. **完整的技术文档交付**——架构文档、API、协议、schema 注册表、故障排除手册
4. **培训交付**——中心技术骨干经过培训，能独立做日常运维和小幅功能调整
5. **合作协议明确退出保护条款**——任何情况下中心积累的数据、证据包、专家分身都归中心所有，与我方无关

**六、最后一句关于'组织稳定'**：

我们的确不是大厂那种看起来巨型稳定的组织。但我们的产品组织是**围绕 15 年持续研究的理论方向建起来的**——这个方向本身的深度 + 我们团队在这个方向上持续投入的一致性，是一种**另一类的稳定性**。

也就是说：
- 大厂的稳定是'**谁走都能顶上**'
- 我们的稳定是'**核心人在这个方向投入 15 年不变**'

两种稳定性各有优劣。对'需要创新、需要持续打磨运行时内核'的合作，后者往往更匹配。

**但具体到合作可操作性**——我们不要求中心相信我们'一定稳定'，而是**通过上面 5 条抗风险机制把'不稳定'的代价降到最低**——任何时候出问题，中心都有明确兜底。这才是真正负责任的合作结构。

## 附录 · 通用应答技巧（现场参考）

### 1. 被问到"技术细节"时

- **可讲**：我们做什么、为什么这么做、给中心带来什么、能出什么证据
- **不讲**：内核算法公式、控制器网络结构、损失函数具体形式、超参数
- **边界话术**："这涉及我们三项独立原创理论方向的具体实现，**签订基础保密协议后可以深度开放**，今天只讲到架构和承诺层。"

### 2. 被问到"跟 XX 大厂某产品是不是一样"时

- **先承认共性**："是，表层看确实有相似之处"
- **再点出差别**："但 XX 在做的是 {从 0 到 1 搭 AI 助手 / 上下文工程 / Agent 编排}，我们在做的是 {AI 进生产后跑长跑稳的运行时底座}"
- **最后落地**："事实上，我们可以叠在他们之上跑，正交不竞争"

### 3. 被问到"你们凭什么"时

- **不卑不亢、事实说话**：科研原创性（杨柳博士 + 论文）、两个领域落地、通义千问上的完整证据包、在签协议后可现场验证
- **留一个开放口**："这四条每一条都可以现场验证、可以查证、可以签字对赌"

### 4. 被问到不知道的问题时

- **不要硬答**。诚实回答：
  "这个问题非常专业，需要我们团队的 {具体角色} 来回答才能准确。我今天可以记下来，下次技术深度交流带着书面答复来。"
- **比编一个答案好一百倍**——中心技术团队在这个领域太专业，硬答会被当场识破。
